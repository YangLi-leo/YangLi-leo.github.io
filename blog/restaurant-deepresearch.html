Restaurant DeepResearch: Creating a Multi-Agent System Inspired by Real-Life Decision-Making via Using MCP Server

April 11, 2025 AI Projects

Restaurant DeepResearch

This project is open-source! Check out the code, contribute, or use it for your own projects.

View on GitHub

The idea for this project came from a painfully familiar situation: trying to decide where to eat. I'd open Google Maps with high hopes, only to be buried in a sea of restaurants that all kind of looked the same. The filters? Not helpful. My cravings? Vague at best. Every choice felt like a gamble. Sometimes I'd text a friend for help, and while the conversation would often start with indecision, it usually turned into something more helpful. We'd share tips, recommend places, and even discover new spots through our back-and-forth. In the end, we almost always found somewhere that worked. That's when it hit me‚Äîwhat if I had a smart assistant who actually understood what I wanted, could think through the options, and maybe even talk it out with me like a foodie friend who never gets tired?

That led to a core question: how can we build an AI system that mimics how humans decide where to eat‚Äîthrough discussion, reasoning, and using external tools like Google Maps? Most existing AI planning systems rely on rigid workflows, yet real-life decision-making is often spontaneous and conversational. This project explores how a multi-agent system can simulate this natural, collaborative process.

To enable this, I leverage an MCP (Model Context Protocol) server to unify various tool-calling APIs and use the CAMEL-AI framework to build agents capable of reasoning, communicating, and acting in a shared environment. The goal is to replicate the casual yet effective way people choose restaurants: search, compare, ask, and refine.

This project is inspired by several ideas:

  1. MCP Servers' Growing Popularity: Nowadays, MCP servers have become very popular, significantly streamlining the process for developers creating tool-calling functions for large language models (LLMs). For example, I use the Google Maps MCP server in this project, combining seven useful tools‚Äîsearch_nearby, get_place_details, maps_geocode, maps_reverse_geocode, maps_distance_matrix, maps_directions, and maps_elevation‚Äîinto a single JSON file, which is remarkably efficient.
  2. Mimicking Real-life Decision-making: When searching for places to eat, I typically use Google Maps and then discuss options with my friends, eventually reaching a satisfying decision. So why not allow AI agents to replicate this same process?
  3. Encouraging Open-ended Exploration: I don't believe we necessarily need a rigid workflow for this type of task. Our successful decisions often result from discussions and interactions with external tools like Google. Creating an environment where agents can autonomously explore ideas is crucial. That's why I chose to use CAMEL-AI's framework instead of alternatives like LangChain or AutoGen.

Based on these ideas, my approach was to break down the task into manageable components or, more accurately, recreate my daily decision-making environment for AI agents.

Step-by-step Approach

Firstly, my project is triggered by user prompts. From a product design perspective, it's unrealistic to expect users to write prompts in markdown or clearly specify every need. Therefore, we require a robust reasoning model to thoroughly understand user intentions and needs.

Secondly, beyond understanding users, models also need to understand each other. Thus, we applied prompt engineering techniques to ensure the output from the first reasoning model is structured clearly in markdown format. This step significantly enhances the task environment, setting the stage effectively for subsequent role-playing sessions.

Thirdly, the project's most crucial component is the role-playing session itself. Since this is my first significant project, I chose a straightforward implementation‚Äîdirectly using CAMEL's pre-defined OwlRolePlaying class. This class involves two roles: a user and an assistant. To conceptualize this, imagine chatting with a friend who happens to be an interactive toolbox. When you share ideas, it executes relevant tools, provides feedback, and reviews its findings‚Äîtruly an impressive capability. Creating such an environment is indeed essential. I'll demonstrate some examples below. For the assistant role, I highly recommend using a powerful tool-calling model.

I also recommend that anyone using this framework take time to carefully read through the conversations between agents. Occasionally, you'll encounter a genuine "Aha!" moment‚Äîwhere the agents demonstrate surprisingly intuitive reasoning or creative insight. These are the sparks that show why this kind of environment matters, and why reinforcement learning could play a key role in making it even better. I'll continue updating this point as the project evolves.

Finally, the sessions are constrained to 10 rounds, ensuring the agents maintain high-quality dialogue within a limited context. The assistant's final answer is the output you receive, clearly explaining why a specific restaurant was chosen while offering engaging, meaningful interactions.

Evaluation Methodology

To evaluate the performance of different tools, I tested the same task across four platforms: Google DeepResearch, Perplexity DeepResearch, OpenAI DeepResearch, and Manus (yes‚ÄîI recently got access). While I don't have a formal benchmark, this comparison offers a practical view of each tool's behavior under the same conditions.

To ensure fairness, I used nearly identical prompts across all tools. Since Manus includes a human-in-the-loop design, I first submitted the original prompt to Manus, then incorporated the additional information it requested into a revised version. This slightly modified version was then used as the standard prompt for the other tools, including my own system. Due to current access limitations, I was only able to use Gemini 2.5 Pro Experimental, which unfortunately hit its quota during testing. I'll update my own results as soon as I'm able to rerun the full test.

Japanese cuisine features wonderful concepts like "omakase" and "kaiseki," both deeply rooted in seasonality. "Omakase," often found in sushi restaurants, means "I'll leave it up to the chef." It's an artistic course where the chef selects the freshest seasonal ingredients of the day to create masterful sushi. "Kaiseki," on the other hand, is a traditional multi-course Japanese dinner, meticulously planned from the seasonal ingredients and beautiful presentation to the harmony with the serving ware and the sequence of dishes. Both offer a profound way to appreciate Japan's rich food culture and seasonal bounty, making them excellent choices for your early October visit.

Beyond Asakusa: Exploring Tokyo's Charming Historic Districts for Food Lovers

First, let's touch upon Asakusa, your benchmark for the desired area. Asakusa is renowned for its historical charm and vibrant atmosphere, home to the iconic Senso-ji Temple and traditional food stalls. However, Tokyo boasts other districts that share this culturally rich, historic ambiance.

Kagurazaka:

Once a prominent geisha district, Kagurazaka, sometimes called "Tokyo's Little Paris," offers a stylish yet distinctly traditional atmosphere with remnants of the Edo period. Its stone-paved alleys host a unique blend of traditional ryotei (high-class Japanese restaurants) and modern eateries. It's a culturally rich area known for high-quality dining.

Nihonbashi & Kyobashi:

Nihonbashi has been a commercial center since the Edo period, blending historical significance with modern sophistication. It's conveniently located near Ginza. Adjacent Kyobashi offers a slightly quieter, more refined atmosphere than Ginza, known for quality dining without the same level of bustle. These areas provide a more mature historical feel than Asakusa, steeped in culinary tradition.

Kanda:

Located near Akihabara but with its own distinct character, Kanda mixes historical elements with modern development. It offers good accessibility and, more recently, has seen the emergence of newer restaurant styles actively welcoming international visitors.

When considering these districts for a family, nuances emerge. Asakusa, while historically charming, tends to offer either very traditional establishments or more casual options. Kagurazaka's blend of tradition and modernity might yield restaurants that are elegant yet relaxed, offering modern comforts. Nihonbashi and Kyobashi, while possessing history and quality, also have strong business district aspects, meaning restaurants might lean towards adult or corporate dining. Kanda, however, offers accessibility and potentially newer establishments more consciously catering to families. Therefore, focusing on areas like Kagurazaka, Kanda, or Kyobashi might increase the chances of finding a restaurant that balances historical ambiance with family-friendly comfort and modern elements.

Conclusion

This project demonstrates the potential of multi-agent systems to simulate human-like decision-making processes. By leveraging MCP servers and the CAMEL-AI framework, I've created a system that can understand complex user needs, reason through options, and provide thoughtful recommendations in a conversational manner.

The comparison with other AI tools highlights both the strengths and areas for improvement in my approach. While Google and OpenAI DeepResearch provide comprehensive, well-structured responses, and Perplexity offers a more concise analysis, my system aims to combine the best of both worlds: detailed information with interactive reasoning.

Future Directions

  1. Enhanced Tool Integration: Expanding the range of tools available to agents, such as reservation systems, review aggregators, and real-time availability checkers.
  2. Improved Multi-turn Conversations: Developing more sophisticated dialogue management to handle follow-up questions and clarifications naturally.
  3. User Preference Learning: Implementing mechanisms for the system to learn and remember user preferences over time, creating more personalized recommendations.
  4. Evaluation Framework: Developing a more rigorous evaluation methodology to assess the quality and usefulness of recommendations.

By continuing to refine this approach, I believe we can create AI systems that not only provide information but truly assist in decision-making processes in a way that feels natural and helpful to users.
